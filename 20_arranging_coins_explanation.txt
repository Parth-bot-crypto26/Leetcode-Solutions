===================================================================
LEETCODE 441: ARRANGING COINS
===================================================================

[PROBLEM DESCRIPTION]
You have 'n' coins and you want to build a staircase where the i-th row 
has exactly 'i' coins. The last row of the staircase may be incomplete.
Given 'n', return the total number of complete rows of the staircase 
that can be built.

-------------------------------------------------------------------
[METHODOLOGY: BINARY SEARCH ON THE ROW RANGE]
-------------------------------------------------------------------

The total number of coins required to completely fill 'k' rows forms an 
arithmetic progression sum, calculated using Gauss's formula: 
Total Coins = k * (k + 1) / 2.

Since the number of possible complete rows is monotonically increasing and sorted 
(from 1 to 'n'), we can perform a Binary Search to find the optimal threshold 'k'.

1. OVERFLOW IMMUNITY (long vs int):
   - Given that 'n' can be as large as 2^31 - 1, computing 'k * (k + 1)' will 
     instantly cause a 32-bit signed integer overflow. To safeguard against 
     undefined behavior, we declare our range pointers and calculation trackers 
     using the 64-bit 'long' (or 'long long') data type.

2. RANGE LOGIC GATES:
   - Calculate the midpoint row: k = left + (right - left) / 2.
   - Calculate the required coins for 'k' complete rows: curr_coins = k * (k + 1) / 2.
   
   - Case A (curr_coins == n): 'k' rows use exactly 'n' coins. We return 'k' immediately.
   - Case B (curr_coins < n): We have leftover coins. It is possible we can build 
     more rows. We shift our search window to the right: left = k + 1.
   - Case C (curr_coins > n): 'k' rows require more coins than we have. This row limit 
     is impossible. We shift our window to the left: right = k - 1.

3. CONVERGENCE TERMINATION:
   - If the loop finishes without an exact match, the 'left' and 'right' pointers 
     cross. At this boundary state, 'right' will always point to the largest integer 
     floor value where the coins required are less than 'n'. Thus, we cast and return '(int)right'.

-------------------------------------------------------------------
[DRY RUN EXAMPLE]
-------------------------------------------------------------------
Input: n = 5 (Expected Output: 2)
Initial State: left = 0, right = 5

Iteration 1:
  - k = 0 + (5 - 0) / 2 = 2
  - curr_coins = 2 * (2 + 1) / 2 = 6 / 2 = 3
  - Check: Is 3 == 5? No. Is 3 < 5? Yes.
  - Action: Shift right: left = k + 1 = 3

Iteration 2:
  - k = 3 + (5 - 3) / 2 = 4
  - curr_coins = 4 * (4 + 1) / 2 = 20 / 2 = 10
  - Check: Is 10 <= 5? No, it's greater.
  - Action: Shift left: right = k - 1 = 3

Iteration 3:
  - k = 3 + (3 - 3) / 2 = 3
  - curr_coins = 3 * (3 + 1) / 2 = 12 / 2 = 6
  - Check: Is 6 <= 5? No, it's greater.
  - Action: Shift left: right = k - 1 = 2

Loop terminates because left (3) > right (2).
Returns (int)right -> 2.

-------------------------------------------------------------------
[COMPLEXITY ANALYSIS]
-------------------------------------------------------------------

Time Complexity: O(log n)
- Explanation: The search boundaries between 1 and 'n' are divided in half during 
  every single iteration of the while loop, leading to a highly optimized logarithmic 
  runtime performance.

Space Complexity: O(1)
- Explanation: The calculation executes entirely in-place. Auxiliary memory 
  allocation is strictly restricted to local scalar variables ('left', 'right', 
  'k', 'curr_coins'), preserving absolute constant space.
